# Черновик: заметки по тестированию и фиксам

Не нормативный документ: наблюдения при ручном или автоматическом тестировании до того, как решения попадут в `spec.md` / `tasks.md` и контракты.

## 009 HTTP Public API — слоты

### `POST /api/v1/slots` — многократные вызовы

**Статус:** открытый вопрос.

**Наблюдение:** после вызова `POST /api/v1/slots` его можно вызвать снова много раз (например, 150) — и столько же раз стартует workflow. Пока не ясно, насколько это желаемое поведение: нужны ли идемпотентность, дедупликация, лимиты, правило «один активный run на слот» и т.п.

**Дальше:** зафиксировать ожидаемую семантику в продукте/архитектуре и при необходимости обновить `specs/009-http-public-api/spec.md` и связанные контракты.

## Pipeline stage 3 — статус строки перезаписывает `PASSED_STAGE_2`

**Статус:** на правки (известное ограничение текущей модели).

**Наблюдение:** при `PersistPipelineStage3Activity` для вакансий, которые были в `PASSED_STAGE_2` и попали в батч скоринга, репозиторий выставляет терминальный статус `PASSED_STAGE_3` или `REJECTED_STAGE_3` (та же колонка/поле статуса на `pipeline_run_jobs`). В результате эти строки **больше не числятся как `PASSED_STAGE_2`**: для списков и фильтров «только прошедшие stage 2» такие записи **пропадают**, хотя по смыслу они stage 2 уже прошли.

**Дальше:** решить продуктово, нужна ли история стадий (отдельные поля/снимки, не перезатирающие stage 2), отдельный список кандидатов stage 3 без смены статуса с `PASSED_STAGE_2`, или иная семантика API — и затем обновить `specs/004-pipeline-stages` / хранилище / эндпоинты.

## Чек-лист: повторные триггеры стадий

- Повторно дернуть **стадию 2** и **стадию 3** (второй/очередной запуск после уже отработавшего пайплайна) — проверить идемпотентность, статусы, батчи и отсутствие сюрпризов в Temporal/UI.
